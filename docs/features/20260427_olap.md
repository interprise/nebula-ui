### OLAP Cube Rewrite — Plan

Date: 2026-04-27. Spans CORE (Java framework) and entrasp-ui (React client). The legacy ExtJS OLAP renderer (`StdHyperCubeRenderer`) is replaced; the in-memory `HyperCube` aggregator on the server is retired.

#### Goals & constraints

- **Heap pressure**: today `OlapCubeDataCursor.populateHypercube()` runs SELECT without `GROUP BY` and aggregates raw fact rows in `HyperCube.mergeMesValues()`, which OOMs on large fact sets. Push aggregation to the database.
- **Cost**: OLAP is a smaller-customer feature; large customers use external MS PowerPivot. **No paid component licenses** (AG Grid Enterprise, Flexmonster, Syncfusion, etc. all ruled out). Custom MIT/Apache or hand-rolled only.
- **Bundle**: lazy-load the OLAP component so non-OLAP customers carry zero cost. AG Grid is already in the main vendor chunk via `ListRenderer`, so reusing it for the OLAP rendering surface adds no library weight — only the OLAP component code splits off.
- **Scroll-ability**: vertical + horizontal scroll with sticky headers and sticky row labels — better than the current renderer.
- **Server-driven rendering** stays (CORE produces the JSON descriptor, React consumes).

#### Architecture

**Server (CORE):**
1. Modify the SELECT-list builder so non-calculated measures wrap in their `Measure.getAggregateType()` SQL function (`SUM`/`COUNT`/`AVG`/`MIN`/`MAX`). Append `GROUP BY` over a supplied dim subset. Default initial fetch uses **all** cube dimensions (`Cube.getAllDimensions()`) so the client receives a full-granularity base set.
2. **NDJSON streaming protocol for the rows-heavy path.** The standard `ServerResponse` envelope carries only a thin placeholder for the OLAP control (`{ "type": "olapCube", "viewName": "olapFatturato" }`). A dedicated endpoint (e.g. `/entrasp/controller/olap`) responds with `Content-Type: application/x-ndjson`, writing each row directly from `populateAggregated`'s callback to the HTTP output stream. Server heap stays O(1) per query — no in-memory row buffer. Client can render header + first rows before the cursor finishes.
   ```
   {"kind":"meta","viewName":"olapFatturato",
    "layout":{"x":["dim0","dim2"],"y":["dim1"],"z":["dim3"]},
    "dimensions":[{"name":"dim0","description":"...","navigateViewName":"...",
                   "values":{"code":"description","...":"..."}}],
    "measures":[
      {"name":"mes0","description":"...","format":"0.00",
       "aggregateType":"SUM","active":true},
      {"name":"margin","description":"Margin","format":"0.00","active":true,
       "calculation":{"expr":"...","ast":{"kind":"op","token":"-","args":[...]}}}],
    "filters":{"dim3":"selectedCode"},"splitCells":false}
   {"kind":"row","d":["c1","c2","c3","c4"],"m":[123.45,null]}
   {"kind":"row","d":["c1","c2","c3","c5"],"m":[67.89,2]}
   ...
   {"kind":"end"}
   ```
   On a mid-stream failure, server emits a final `{"kind":"error","message":"..."}` line; client treats partial data as failed. Every line is self-describing via `kind`. Tuples (`d`, `m`) are positional to keep payload size tight; client correlates against `dimensions`/`measures` order from the meta line.
3. Calculated-measure ASTs serialized from `ParseNode` trees (operators `+ - * /`, unary `-`, `round`, `Atom` = measure name, `NumberConstant`).
4. XLS/CSV exports re-run via `populateAggregated()` and apply the client-supplied X/Y/Z layout.
5. Drill-through (`NavigateOlapCommand`) is unchanged.
6. Legacy HTML path (`renderHTML`, `StdHyperCubeRenderer`, `HyperCube`, `AlterCubeCommand`) deleted at the end after parity verification.

**Client (entrasp-ui):**
- Lazy-loaded chunk `src/components/olap/OlapCubeRenderer.tsx` (`React.lazy` + `<Suspense>`, mirror of `Shell.tsx:200` `CdmsTree` pattern). Registered as `'olapCube'` in `EXPECTED_CONTROL_TYPES`.
- **`useOlapState(payload)`** owns: per-dim position (X/Y/Z), per-dim and per-measure visibility, dim ordering within axis, Z-filter values, splitCells, and (#12) per-Y-tree-node expand state.
- **Base dataset** kept in memory at full cube-dim granularity from the initial fetch and never overwritten by visibility changes.
- **Display computation**: project from base — for each visible-dim combo (or partial dim-prefix path for collapsed parents), fold measures across hidden buckets. Composable measures (SUM/MIN/MAX/plain COUNT) project correctly; non-composable (AVG, COUNT DISTINCT, MEDIAN, STDDEV, percentiles) only project correctly when no dims are hidden / no parents are collapsed.
- **Server refetch trigger** (only round-trip beyond initial + drill-through): a non-composable measure is visible AND some dim is hidden / parent collapsed. Refetch with current visible-dims subset, store as override alongside the base; restoring a composable-only display drops the override.
- **Rendering surface: AG Grid Community** (already in the vendor chunk):
  - Y dims → pinned-left columns with custom cell renderers (indentation + chevron for expand/collapse).
  - X dim hierarchy → nested `columnDefs` `children` (multi-level pinned-top header). Collapsible parents via native `columnGroupShow: 'open' | 'closed'`.
  - Cell columns: one per (X-leaf × visible measure) with `valueGetter` reading the projected cell index. Numeric formatting via `valueFormatter` per measure's `format`.
  - Row + column virtualization, smooth two-axis scroll, sticky headers/cols — all free.
- Z-filters / legend / pivot arrows / measure toggles / split-cells live outside the grid in a side or top panel.
- **Calc evaluator** `src/components/olap/calc.ts` ports `CalculationVisitor.java` to a stack-based JSON-AST walker; `null` propagates the same way (`null + x = null`).
- **Drill-through**: header-cell click → `api.postAction('olap.NavigateOlap', { navpath, option1 })`.
- **Exports**: button → `api.triggerDownload('XLSPivotCommand'|'OlapCSVCommand', { ...layout, visibleDims })` so the server formats against current pivot state.

#### Composability rules for projection

| `aggregateType` | Composable (re-project from full-granularity base) |
|---|---|
| SUM | yes |
| MIN | yes |
| MAX | yes |
| COUNT (plain row count) | yes |
| AVG | no — needs server refetch when display granularity drops |
| COUNT DISTINCT, MEDIAN, percentiles, STDDEV | no — same |

Almost all production measures are SUM. The non-composable refetch path is rare.

#### Tasks

1. CORE: SQL aggregation in `OlapCubeDataCursor` (+ `populateAggregated(visibleDims)`)
2. CORE: `ParseNode` → JSON serialization for calculated-measure ASTs
3. CORE: `OlapCubeUIControl.renderJSON()` emits a thin placeholder descriptor; add a dedicated NDJSON streaming endpoint (`/controller/olap`) that resolves the cube + view from request params and streams `meta` + `row` + `end` lines directly through the HTTP response writer. Same path handles initial fetch and refetch (with `visibleDims` parameter)
4. CORE: XLS/CSV exports consume `populateAggregated()` and the client's current layout

6. UI: `OlapCubeRenderer` + `useOlapState` + projected cell index from base dataset
7. UI: AG Grid Community as rendering surface (pinned-left Y dims, nested column groups for X, valueGetter cells)
8. UI: calculated-measure evaluator (JS port of `CalculationVisitor`)
9. UI: legend, Z-filter combos, measure toggles, pivot/reorder arrows, splitCells
10. UI: drill-through + exports + conditional refetch wiring
11. Verify all 18 entrasp `olap*.xml` views; retire legacy HTML path (`renderHTML`, `StdHyperCubeRenderer`, `HyperCube`, `AlterCubeCommand`)
12. UI: collapsible parent dims on X (native column-group toggle) and Y (hand-rolled tree expand state) — depends on #6, #7

#### Deferred / open

- **Calc operator audit**: cube/model definitions live in DB tables, not source — direct grep impossible. Calculations are rare; add operators to (#2) and (#8) as encountered.
- **Z-filter cardinality**: high-cardinality Z dims could bloat payload. Defer; revisit if real cubes hit the limit.
- **`splitCells` semantics**: not remembered; lift behavior from `StdHyperCubeRenderer` when implementing #9 and decide whether to keep.
- **Very large cubes at full granularity**: tradeoff between payload size and zero-round-trip projection. Defer; if a cube proves too large to ship at full granularity, add a per-cube "lazy dims" hint that excludes selected dims from the initial GROUP BY (each visibility toggle on those would refetch).
- **Branches**: working branch `olap-rewrite` cut from `react-ui` (CORE) and from `master` (entrasp-ui).
